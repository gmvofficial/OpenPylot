# Publishing OpenPylot

Step-by-step guide for releasing OpenPylot to all five package registries.

This document assumes the repo is already prepared (consistent name, license, and
URLs) and that the package builds and tests pass locally â€” that work is done.
Your job here is to **publish** the existing artifacts.

## Recommended order

Do them in this order â€” easiest first, and Homebrew last because it depends on a
GitHub Release existing:

1. **PyPI** (Python) â€” practice on TestPyPI first
2. **npm** (Node.js)
3. **crates.io** (Rust)
4. **Docker Hub**
5. **Homebrew** (needs a GitHub Release with binaries to already exist)

## 1. PyPI (Python package)

The Python package is a **compiled Rust extension** built with
[maturin](https://www.maturin.rs/). It must be built against **Python â‰¤ 3.13**
(PyO3 0.22 does not support 3.14 yet).

### Build & test locally first

```bash
# Use Python 3.13 (NOT 3.14)
python3.13 -m venv .venv313
source .venv313/bin/activate
pip install --upgrade maturin

cd python
maturin develop                 # compiles + installs into the venv
python -c "import pylot; print(pylot.__version__)"   # -> 0.3.0
```

### Publish to TestPyPI (do this first â€” it's the safe sandbox)

```bash
cd python
# Build wheel + source distribution
maturin build --release

# Upload to TestPyPI (token from https://test.pypi.org/manage/account/token/)
maturin upload --repository testpypi target/wheels/*
```

Verify it installs from TestPyPI:

```bash
pip install --index-url https://test.pypi.org/simple/ openpylot
```

### Publish to real PyPI

```bash
cd python
maturin build --release
maturin upload target/wheels/*   # token from https://pypi.org/manage/account/token/
```

> **Note on platforms:** `maturin build` produces a wheel only for the OS/CPU you
> run it on (e.g. macOS-arm64). To serve Linux/Windows/Intel users you need
> [`maturin-action`](https://github.com/PyO3/maturin-action) in GitHub Actions to
> build wheels for every platform. See [CI](#multi-platform-builds-ci).

## 2. npm (Node.js package)

The Node package is a native addon built with
[napi-rs](https://napi.rs/). The entry files (`node/index.js`, `node/index.d.ts`)
are generated by napi and committed to the repo.

### Build & test locally first

```bash
cd node
npm install
npm run build                    # produces openpylot.<platform>.node

# Verify the package contents before publishing (uploads nothing)
npm publish --dry-run

# Optional: install the tarball into a temp project to confirm it loads
npm pack
```

### Publish

```bash
cd node
npm login                        # browser login to npmjs.com
npm publish --access public
```

Verify:

```bash
npm install openpylot            # in a scratch folder
node -e "console.log(Object.keys(require('openpylot')))"
```

> **Note on platforms:** `npm run build` builds the `.node` binary for your
> platform only. The `optionalDependencies` in `package.json`
> (`openpylot-darwin-arm64`, `openpylot-linux-x64-gnu`, â€¦) are the per-platform
> sub-packages that `napi prepublish` publishes from CI. Until CI exists, the
> published package works for the platform it was built on. See
> [CI](#multi-platform-builds-ci).

## 3. crates.io (Rust crate)

The simplest one â€” crates.io builds from source.

### Test locally first

```bash
# Dry run: packages the crate and checks it, uploads nothing
cargo publish --dry-run
```

### Publish

```bash
cargo login <token>              # token from https://crates.io/settings/tokens
cargo publish
```

Verify at `https://crates.io/crates/openpylot`. Users then install with:

```bash
cargo install openpylot          # installs the `pylot` binary
```

## 4. Docker Hub

A multi-stage image (Rust builder â†’ slim Debian runtime), runs as non-root.

### Build & test locally first

```bash
docker build -t openpylot:0.3.0 .
docker run --rm --entrypoint pylot openpylot:0.3.0 --version   # -> pylot 0.3.0
```

### Publish (single architecture â€” quick)

```bash
docker login
docker tag openpylot:0.3.0 globalmindventures/openpylot:0.3.0
docker tag openpylot:0.3.0 globalmindventures/openpylot:latest
docker push globalmindventures/openpylot:0.3.0
docker push globalmindventures/openpylot:latest
```

### Publish (multi-architecture â€” recommended for real users)

This serves both Intel/AMD (`amd64`) and Apple-Silicon/ARM servers (`arm64`):

```bash
docker buildx create --use   # once
docker buildx build --platform linux/amd64,linux/arm64 \
  -t globalmindventures/openpylot:0.3.0 \
  -t globalmindventures/openpylot:latest \
  --push .
```

> The Dockerfile sets `CARGO_PROFILE_RELEASE_LTO=thin` so the build fits within a
> ~3â€“4 GB Docker memory limit. If you build in CI with more RAM you can remove it.

## 5. Homebrew (macOS / Linux CLI)

Homebrew is different: it does **not** build the code, it downloads a
**pre-built binary** from a GitHub Release. So the steps are: build binaries â†’
make a GitHub Release â†’ put their checksums in the formula â†’ host the formula in
a tap repo.

### Step 5a â€” Build the binaries (one per platform)

On each platform (or in CI), build and package the `pylot` binary:

```bash
cargo build --release --bin pylot
mkdir -p dist
tar -czf dist/pylot-darwin-arm64.tar.gz -C target/release pylot   # name per platform
shasum -a 256 dist/pylot-darwin-arm64.tar.gz                      # note the checksum
```

Required tarball names (the formula expects these exact names):

- `pylot-darwin-arm64.tar.gz`   (Apple Silicon Mac)
- `pylot-darwin-x86_64.tar.gz`  (Intel Mac)
- `pylot-linux-arm64.tar.gz`    (ARM Linux)
- `pylot-linux-x86_64.tar.gz`   (x86 Linux)

> You can only build a platform's binary **on that platform** (or by
> cross-compiling). In practice, build all four in CI. See [CI](#multi-platform-builds-ci).

### Step 5b â€” Create the GitHub Release

```bash
gh release create v0.3.0 dist/*.tar.gz \
  --repo globalmindventures/OpenPylot \
  --title "v0.3.0" --notes "OpenPylot 0.3.0"
```

(Or upload the tarballs via the GitHub Releases web UI.)

### Step 5c â€” Put the real checksums in the formula

Edit `Formula/openpylot.rb` and replace each `PLACEHOLDER_SHA256_*` with the
`shasum -a 256` value of the matching tarball.

> The `darwin-arm64` checksum is already filled in. The other three are still
> `PLACEHOLDER_*` and must be replaced once their binaries are built.

### Step 5d â€” Create the tap repo and add the formula

```bash
# Create a repo literally named "homebrew-tap" in the org
gh repo create globalmindventures/homebrew-tap --public

# Add the formula to it
git clone https://github.com/globalmindventures/homebrew-tap
cp Formula/openpylot.rb homebrew-tap/
cd homebrew-tap && git add openpylot.rb && git commit -m "Add openpylot 0.3.0" && git push
```

### Step 5e â€” Users install with

```bash
brew tap globalmindventures/tap
brew install openpylot
```

## Multi-platform builds (CI)

PyPI wheels, npm `.node` binaries, Homebrew tarballs, and Docker images all need
to be built for **multiple OS/CPU combinations**. A laptop can only build for its
own platform. The production solution is **GitHub Actions** with a matrix of
macOS + Linux (+ Windows) runners that build every artifact on a `v*` tag and
publish them automatically.

Recommended workflows:

- **PyPI:** [`PyO3/maturin-action`](https://github.com/PyO3/maturin-action)
- **npm:** napi-rs provides a CI template via `napi prepublish`
- **Homebrew:** a release workflow that builds all four binaries and attaches them
- **Docker:** `docker/build-push-action` with `platforms: linux/amd64,linux/arm64`

> This is the next milestone after the first single-platform publishes succeed.

## Releasing a new version

For every new release, bump the version in **all** of these, keeping them in sync:

- `Cargo.toml` (`version`)
- `python/pyproject.toml` and `python/Cargo.toml`
- `node/package.json` and `node/Cargo.toml`
- `Formula/openpylot.rb` (`version`)
- README badge / `__version__` strings

Then: tag the release (`git tag v0.x.y && git push --tags`), rebuild artifacts,
and re-run the publish steps above. Registries never allow overwriting an
existing version, so a mistake means publishing a new patch version.

## Pre-publish checklist

- [ ] Version bumped consistently across all manifests
- [ ] `cargo publish --dry-run` passes
- [ ] `maturin build --release` produces a wheel
- [ ] `npm publish --dry-run` shows the correct files
- [ ] `docker build` succeeds and `pylot --version` runs in the container
- [ ] No secrets committed (`.env` is gitignored â€” never `git add -f` it)
- [ ] You have publish access to the `globalmindventures` org on each registry
